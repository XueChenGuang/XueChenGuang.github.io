# 缓存同步策略

https://juejin.cn/post/7024825393018306574
吹爆，这篇写的真的很好。

知识输出的时候到了，看了上面的文章之后，思路通达了很多。

对于缓存同步策略，也就是为了解决 mysql+redis 之后数据不一致的问题。这个一致和事务一致有些区别，但是在思想上还是差不多的，都有涉及到 cp，ap 思想。

展开说的话就是：

  • 更新，先更新数据库还是 Redis 呢？更新的好处就是不会出现缓存打空的情况。
  • 删除，先删 Redis 还是先更数据库呢？

更新的话，无论是先更新数据库还是先更新缓存都会出现多请求而出现数据不一致的情况 A 更 1，B 更 2

  • A 更数据库 1
  • B 更数据库 2
  • B 更缓存 2
  • A 更缓存 1
    最终就是 1，同理选择先更缓存再更数据库也是一样。相比 read，write 更耗时，所以比较频繁出现上面的情况。

删除的话，就是 A 请求删，B 请求读。

  • （之前是 1）A 删缓存
  • B 读缓存（没有），读数据库（1）
  • B 将 1 写入到缓存
  • A 将数据库写为 2
    那么就造成了数据库和缓存不一致了，先删缓存有改进策略就是延迟双删，也就是更改的这个请求的后端加上一个 sleep，确保同时间有读的一定可以读完并且更改了缓存，所以无论是读到旧数据还是新数据，后面都再加一个删除 redis，这样下次有请求来的话，肯定就是再读数据库，这样肯定就是新值了。

如果是先更数据库，然后删缓存的情况。

  • A 读缓存，缓存中不存在，此时去读数据库
  • A 读到 1，准备写入缓存
  • B 修改数据库，改为 2
  • B 删除缓存
  • A 将 1 写入缓存
    出现不一致，可见上面 A 的读取写缓存要涵盖整个 mysql 的 update 以及删缓存，这个情况基本不会出现，概率很小，毕竟 update 的耗时是远大于 read 的。

以上的情况都是建立再 mysql 和 redis 都成功的情况下，毕竟是两个服务，从实现来说的话就是两个进程，这个是没有事务的，如果要实现一致性，也就是需要确保一个成功另一个必然成功的情况，来了
1、重试，确实是我 mysql 更新完了之后，redis 删除失败，重试删除
2、同步还是异步，同步的话就是强一致，也就是 cp。从同步的角度来讲就是严重影响吞吐量，因为这一条请求就会一直在，知道 redis 删除成功才结束。
3、使不使用中间件，也就是异步是如何实现的，肯定有一个中间件在充当着重发的这个角色，那么就是 mq 了
4、对于异步的话，是直接使用 mq 来接收 redis 的删除请求，然后不停的重试直到成功，还是使用 canal（阿里开源）模拟从库请求主库的 binlog，只要 mysql 写入成功，那么就有 binlog，那么 canal 就会拿到信息从而完成下游的任务发给 mq，保证了消息生产的不丢失。
